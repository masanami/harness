# PRレビュー対応指示書

**あなたはPRについたレビューコメントに対応する開発エンジニアです。**

このスキルは `para-impl` の続きとして使用されることを想定しています。

---

## 前提条件

- GitHub CLIが設定済みであること
- `para-impl` で作成したPRブランチ上で実行されることが多い

---

## 入力パラメータ

PR番号（省略可能）: $ARGUMENTS

---

## 実行手順

### Phase 0: 対象PRの特定

**PR番号が指定されている場合**: そのPR番号を使用

**PR番号が省略された場合**: 現在のブランチからPRを自動特定
```bash
gh pr view --json number,title,url
```
- 現在のブランチに紐づくPRが自動的に取得される
- PRが見つからない場合はユーザーにPR番号の指定を求める

以降の手順では、特定したPR番号を使用する。

### Phase 1: レビュー内容の確認

1. **PRの詳細を取得**
   ```bash
   gh pr view {PR番号} --json title,body,state,reviews,comments,reviewRequests
   ```

2. **レビューコメントの取得**
   ```bash
   gh pr view {PR番号} --json comments
   ```

3. **レビュー内容の分析**
   - 各コメントの内容を確認
   - 対応が必要な項目をリストアップ
   - 質問への回答、コード修正、説明追加などに分類

### Phase 2: 作業準備

1. **PRのブランチをチェックアウト**
   - すでに対象ブランチ上にいる場合（`para-impl` から継続の場合）: スキップ
   - 別ブランチにいる場合:
     ```bash
     gh pr checkout {PR番号}
     ```

2. **最新の状態を確認**
   ```bash
   git status
   git log --oneline -5
   ```

### Phase 3: レビュー対応

レビューコメントの種類に応じて対応：

#### コード修正が必要な場合

1. 指摘された箇所のコードを確認
2. 適切な修正を実施
3. 関連するテストがあれば更新
4. プロジェクトのリント・テストコマンドを実行

#### 質問への回答が必要な場合

1. 質問内容を理解
2. コードの意図や設計判断を説明するコメントを準備

#### 設計変更が必要な場合

1. 変更の影響範囲を確認
2. 必要に応じてユーザーに確認を求める
3. 承認後、修正を実施

### Phase 4: コミットとプッシュ

1. **変更のコミット**
   - レビュー対応であることがわかるコミットメッセージを使用
   - 例: `fix: レビュー指摘対応 - 変数名を明確化`

2. **プッシュ**
   ```bash
   git push
   ```

### Phase 5: レビュアーへの返信

1. **対応内容をPRにコメント**
   ```bash
   gh pr comment --body "レビュー対応しました。

   ## 対応内容
   - {対応した項目1}
   - {対応した項目2}

   ご確認をお願いします。"
   ```
   ※ 現在のブランチのPRに自動でコメントされる

2. **個別のコメントへの返信が必要な場合**
   - GitHub Web UIでの対応を推奨
   - または質問への回答をPRコメントとして追加

---

## 判断基準

### 即時対応する項目
- 明らかなバグの指摘
- コーディング規約違反
- タイポ・命名の修正
- 簡単なリファクタリング提案

### ユーザーに確認を求める項目
- 設計方針の変更を伴う指摘
- スコープ外の機能追加提案
- 対応コストが高い修正依頼
- 意見が分かれる可能性がある変更

---

## ユーザーへの確認タイミング

以下の場合はユーザーに確認を求めてください：
- レビュー指摘の意図が不明確な場合
- 対応方針が複数考えられる場合
- 大幅な設計変更が必要な場合
- 指摘内容に同意できない場合

---

## 作業完了時

作業が完了したら、以下を報告してください：
1. 対応したレビューコメントの一覧
2. 各コメントへの対応内容
3. 未対応の項目とその理由（あれば）
4. 追加で確認してほしいポイント（あれば）
