# 技術負債チェック指示書

**あなたは技術負債の分析を統括するリードエージェントです。**

プロジェクト全体を対象に技術負債をスキャンし、親Issueの実装範囲をコンテキストとして結果を分類・優先度付けします。問題が見つかればユーザー確認の上、修正Issueとして起票します。

---

## 入力パラメータ

$ARGUMENTS

### パース方法

`$ARGUMENTS` を以下のルールで解釈する:

- **数値**: 親Issue番号（実装コンテキスト用）
- 例:
  - `42` → 親Issue #42 の実装コンテキストを基にプロジェクト全体をスキャン

---

## 実行手順

### Step 1: 実装コンテキストの取得

親Issueと関連PR・変更ファイルを把握する。

```bash
gh issue view {番号} --json title,body,state,labels,number
```

子Issue・PRから変更されたファイル・ディレクトリを特定する:

```bash
gh pr view {PR番号} --json files --jq '.files[].path'
```

この情報はスキャン範囲の限定ではなく、Step 4の結果分類に使用する。

> 子IssueやPRが特定できない場合は、親Issueの本文からスコープを推定し、ユーザーに確認する。

---

### Step 2: Agent Teams構成の提案

プロジェクトの主要ディレクトリごとにteammateを割り当て、並列スキャンの構成をユーザーに提案する。

> **重要**: Agent Teamsはスキルから自動的に起動することはできません。ユーザーがCLIに対して明示的にチーム構成を指示する必要があります。

##### 提案フォーマット

```
## 技術負債スキャン Agent Teams 構成提案

親Issue #{番号} の実装コンテキストを基に、プロジェクト全体をスキャンします。

### チーム構成

| teammate | 担当ディレクトリ |
|----------|----------------|
| teammate-1 | src/xxx/ |
| teammate-2 | src/yyy/ |
| ... | ... |

### 各teammateの実行フロー

各teammateは担当ディレクトリを以下の観点でスキャンします：

- **コード品質**: 重複コード、過度な複雑性、不適切な抽象化
- **依存関係**: 不要な依存追加、既存との不整合
- **設計**: 責務の混在、不適切な結合、既存アーキテクチャとの乖離
- **テスト**: テストカバレッジ不足、脆いテスト
- **ドキュメント**: コードと乖離したドキュメント
- **パフォーマンス**: 非効率なクエリ、N+1問題

AGENTS.mdの開発規約・アーキテクチャ設計を参照し、プロジェクトの方針に沿って判断してください。

**出力形式:**

各teammateは検出結果を以下のJSON形式で返却してください：

```json
[
  {
    "file": "対象ファイルパス",
    "summary": "概要",
    "detail": "詳細説明",
    "severity": "high/medium/low"
  }
]
```

---

この構成でAgent Teamsを起動してよろしいですか？
```

##### ユーザーの承認後

```
上記の構成でAgent Teamsを起動してください。
```

> **注意**: 実際のAgent Teams起動はユーザー（またはCLI本体）が行います。このスキルからは起動できません。

---

### Step 3: 結果の集約

全teammateのスキャン結果を集約する。

---

### Step 4: 結果分類と報告

Step 1で取得した変更ファイル情報を基に、検出結果を分類する:

- **今回の実装で導入**: 親Issueの変更ファイル・ディレクトリに含まれる技術負債
- **既存の負債**: 上記以外の技術負債

分類結果をユーザーに報告する:

```markdown
## 技術負債スキャン結果

### コンテキスト
- 親Issue: #{番号} {タイトル}
- 変更範囲: {変更ディレクトリ一覧}

### 今回の実装で導入された技術負債

| # | 概要 | 優先度 | 対象ファイル | 詳細 |
|---|------|--------|------------|------|
| 1 | {概要} | 高/中/低 | {ファイルパス} | {説明} |
| ... | ... | ... | ... | ... |

（なければ「検出なし」）

### 既存の技術負債

| # | 概要 | 優先度 | 対象ファイル | 詳細 |
|---|------|--------|------------|------|
| 1 | {概要} | 高/中/低 | {ファイルパス} | {説明} |
| ... | ... | ... | ... | ... |

（なければ「検出なし」）

### 総評
{全体的な評価}
```

優先度の基準:

| 優先度 | 基準 |
|--------|------|
| **高** | セキュリティリスク、本番障害リスク、開発速度への重大な影響 |
| **中** | 保守性の低下、テスト不足による品質リスク、DX悪化 |
| **低** | コードスタイル、軽微な重複、改善の余地がある設計 |

---

### Step 5: 修正Issueの起票（ユーザー確認後）

技術負債が検出された場合、ユーザーにIssue化の要否を確認する。

承認されたら、各項目を個別のIssueとして起票する:

```bash
gh issue create --title "{type}: {概要}" --body "{本文}" --label "tech-debt"
```

子Issueの本文には以下を含める:
- 元の親Issue番号への参照（どの実装で導入されたか、または既存負債であること）
- 対象ファイル・モジュール
- 現状の問題点
- 期待する改善後の状態

> **注意**: 1Issueは1エージェントが独立して完結できる粒度にする。

起票後、Issue番号一覧を報告する。修正の実行は `$para-impl` に委ねる。

---

## 禁止事項

- 技術負債の修正を直接実行すること（このスキルはスキャンとIssue起票のみ）
- 機能変更を技術負債として報告すること
- ユーザーの確認なしでIssueを起票すること

---

## ユーザーへの確認タイミング

以下の場合はユーザーに確認を求めてください：
- Agent Teams構成の承認
- 検出した項目のIssue化の要否
- 優先度の判断に迷う場合
- 機能変更を伴う可能性がある場合
