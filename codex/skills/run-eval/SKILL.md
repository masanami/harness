# 評価実行スキル

指定されたスキルの評価シナリオを実行し、合格基準に基づいて合否を判定・記録します。

## 入力パラメータ

```text
$ARGUMENTS
```

- 第1引数: スキル名（例: `self-review`, `para-impl`, `reduce-debt`）
- 第2引数: TC番号（例: `TC-01`）。省略時は一覧を表示して選択を促す

---

## 実行手順

### Phase 1: シナリオの読み込み

1. `eval/scenarios/{スキル名}.md` を読み込む
2. TC番号が指定されていない場合は利用可能なケース一覧を表示し、選択を促す
3. 対象TCの内容（前提・入力・期待する動作・合格基準・チェック方法）を確認する

### Phase 2: 環境セットアップ

対象TCの「前提」セクションを読み、必要な環境を整える手順を案内する。

#### self-review のセットアップ案内例

| TC | セットアップ手順 |
|----|----------------|
| TC-01 | `git checkout -b feature/eval-self-review-tc01 main` を実行し、`src/auth.js` が差分に含まれることを確認 |
| TC-02 | `git checkout -b feature/eval-self-review-tc02 main` を実行し、`src/utils.js` に小さなクリーンな変更を加えてコミット |
| TC-03 | `git checkout -b feature/eval-self-review-tc03 main` を実行し、変更を加えずに空の差分状態にする |
| TC-04 | `git checkout -b feature/eval-self-review-tc04 main` を実行し、`src/auth.js` が差分に含まれることを確認 |
| TC-05 | TC-01 と同様の環境を使用 |

#### para-impl のセットアップ案内例

| TC | セットアップ手順 |
|----|----------------|
| TC-01 | main ブランチ上で評価用 Issue #1（`feat: add formatDate function`）が存在することを確認 |
| TC-02 | main ブランチ上で Issue #1, #2 の両方が存在することを確認 |
| TC-03 | main ブランチ上で Issue #3（`feat: add user authentication`）が存在することを確認 |
| TC-04 | 存在しない Issue 番号（例: `99999`）を使用 |
| TC-05 | Issue #1 を使用（単一ファイル変更のみを求める内容） |

#### reduce-debt のセットアップ案内例

| TC | セットアップ手順 |
|----|----------------|
| TC-01 | main ブランチ上で Issue #4（技術負債調査）が存在し、`src/legacy.js` が main にあることを確認 |
| TC-02 | TC-01 と同様 |
| TC-03 | 技術負債のないブランチ状態（`src/utils.js` のみ変更）で実行 |
| TC-04 | main ブランチ上で Issue #4 が存在し、`src/auth.js`（セキュリティリスク）・`src/legacy.js`（重複コード・マジックナンバー）が main にあることを確認 |

セットアップが完了したかユーザーに確認する。

### Phase 3: スキル実行

1. 対象TCの「入力」セクションに記載されたコマンドを提示する
2. コマンドを提示し、実行前にユーザーへ明示確認を取る
3. ユーザー同意後にのみ実行する（未同意なら実行しない）
4. スキルの実行が完了するまで待つ

### Phase 4: 合否判定

対象TCの「合格基準」を1項目ずつ確認する。

各項目について以下の形式で確認:

```text
[チェック {番号}/{合計}]
基準: {合格基準の内容}

「チェック方法」にコマンドがある場合はここで実行する。

この基準を満たしていますか？
  P = Pass（満たしている）
  F = Fail（満たしていない）
  S = Skip（確認できない・該当しない）
```

全項目の確認が終わったら判定サマリーを表示:

```text
## 判定サマリー

| # | 基準 | 結果 |
|---|------|------|
| 1 | {基準1} | ✅ Pass |
| 2 | {基準2} | ❌ Fail |
| 3 | {基準3} | ⏭️ Skip |

### 総合判定: PASS / FAIL
（全項目 Pass または Skip → PASS、1項目でも Fail → FAIL）
```

### Phase 5: 結果の記録

以下のパスに結果ファイルを作成する:

```text
eval/results/{YYYY-MM-DD}/{スキル名}-{TC番号}.md
```

ファイル内容:

```markdown
# {スキル名} {TC番号} 評価結果

## 評価環境
- 実施日: {YYYY-MM-DD}
- 対象バージョン: {git rev-parse HEAD の出力}
- 評価者: （記入してください）

## 総合判定: PASS / FAIL

## 結果サマリー

| # | 合格基準 | 結果 | 備考 |
|---|---------|------|------|
| 1 | {基準1} | ✅ Pass | {所見} |
| 2 | {基準2} | ❌ Fail | {失敗理由} |

## 所見・改善提案

{評価中に気づいた点、スキルへの改善提案など}
```

結果ファイルを作成したら、パスと総合判定をユーザーに報告して完了とする。
