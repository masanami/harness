# 候補評価・ブラッシュアップ

複数の実装候補ブランチを比較評価し、選定された候補に不採用候補の優れた点を取り込んでブラッシュアップする。

単体で使用可能。実装ブランチのコード比較に特化。

---

## 入力パラメータ

候補ブランチ名とオプション: $ARGUMENTS

### パース方法

`$ARGUMENTS` を以下のルールで解釈する:

- **ブランチ名**: スペース区切りで複数の候補ブランチ名
- **`--issue N`**: 対象Issue番号（省略時はブランチ名から推定）
- **`--auto`**: 自動選定モード。ユーザー確認なしで推奨候補を自動採用する
- 例:
  - `feature/issue-1-candidate-1-auth feature/issue-1-candidate-2-auth --issue 1`
  - `feature/issue-1-candidate-1-auth feature/issue-1-candidate-2-auth --issue 1 --auto`

---

## 選定モード

| モード | 動作 | 使用場面 |
|-------|------|---------|
| 対話モード（デフォルト） | 比較レポート → ユーザー選定 → ブラッシュアップ | `$evaluate-candidates` 直接起動 |
| 自動モード（`--auto`） | 比較レポート → 推奨候補を自動採用 → ブラッシュアップ | 自動化パイプラインからの呼び出し |

> **前提**: 候補ブランチはセルフレビュー済み（`self-review` スキル適用済み）であること。呼び出し元が候補実装時にセルフレビューを実施する責務を持つ。

以下のプロンプトを渡すこと（`{{ISSUE_NUMBER}}`, `{{CANDIDATE_BRANCHES}}`, `{{CANDIDATE_RESULTS}}` を実際の値に置換）:

## 候補情報

{{CANDIDATE_RESULTS}}

## 手順

1. 各候補ブランチのコードを `git diff origin/main...{ブランチ名}` で確認する
2. 以下のテンプレートに沿って比較レポートを作成する
3. 各候補の設計・品質・保守性を評価し、推奨候補を1つ選定する

## 比較レポートテンプレート

```markdown
# 候補比較レポート

## 対象Issue
- Issue: #{issue_number} {issue_title}

## 候補一覧
| # | ブランチ | ステータス | テスト結果 | 変更行数 (+/-) |
|---|---------|-----------|-----------|---------------|
| 1 | `{ブランチ名}` | success/partial/failed | pass/fail | +XX / -YY |
| 2 | `{ブランチ名}` | success/partial/failed | pass/fail | +XX / -YY |

## 各候補の分析
### 候補 N: {アプローチ名}
**ブランチ**: `{ブランチ名}`
**アプローチ概要**: {説明}
**良い点**: ...
**懸念点**: ...
（候補数に応じて繰り返す）

## 比較テーブル
| 観点 | 候補 1 | 候補 2 | ... |
|------|--------|--------|-----|
| **設計の明瞭さ** | {評価} | {評価} | ... |
| **コードの可読性** | {評価} | {評価} | ... |
| **テストカバレッジ** | {評価} | {評価} | ... |
| **パフォーマンス** | {評価} | {評価} | ... |
| **保守性** | {評価} | {評価} | ... |
| **変更の最小性** | {評価} | {評価} | ... |
| **総合評価** | {S/A/B/C} | {S/A/B/C} | ... |

## 推奨
**推奨候補**: 候補 {N}
**理由**: {推奨理由の詳細説明}
```

## 出力

テンプレートに沿った比較レポート全文をそのまま出力すること。最終行に以下のJSON形式で推奨候補番号を追記すること:

```json
{ "recommended_candidate": 推奨候補番号 }
```

### Phase 2: 候補選定

#### 対話モード（デフォルト）

1. Phase 1で生成された比較レポートをユーザーに提示する
2. ユーザーに候補の選択を求める
3. 選定結果を記録し、Phase 3（ブラッシュアップ）へ進む

#### 自動モード（`--auto`）

1. Phase 1の比較レポートから `recommended_candidate` を取得する
2. 推奨候補を自動採用し、Phase 3（ブラッシュアップ）へ進む

> **注意**: この時点ではPR作成・ブランチ削除はまだ行わない。呼び出し元がPR作成を担当する。

---

### Phase 3: ブラッシュアップ

選定された候補に、不採用候補の優れた点を取り込んで品質を向上させる。

Task toolを以下の設定で起動する:

- **subagent_type**: `general-purpose`

以下のプロンプトを渡すこと（`{{ISSUE_NUMBER}}`, `{{SELECTED_BRANCH}}`, `{{SELECTED_CANDIDATE}}`, `{{REJECTED_BRANCHES}}`, `{{COMPARISON_REPORT}}` を実際の値に置換）:

````
あなたは実装ブラッシュアップエージェントです。選定された実装候補に、不採用候補の優れた点を取り込んで改善してください。

## コンテキスト

- Issue: #{{ISSUE_NUMBER}}
- 選定候補: candidate-{{SELECTED_CANDIDATE}}（ブランチ: {{SELECTED_BRANCH}}）
- 不採用候補ブランチ: {{REJECTED_BRANCHES}}

## 比較レポート

{{COMPARISON_REPORT}}

## 手順

### Step 1: プロジェクト理解

プロジェクトルートの `AGENTS.md` を読み、開発規約・テスト方針・よく使うコマンドを把握する。

### Step 2: 選定候補のコード確認

```bash
git checkout {{SELECTED_BRANCH}}
git diff origin/main...{{SELECTED_BRANCH}}
```

選定候補の実装全体を把握する。

### Step 3: 不採用候補の良い点を抽出

各不採用候補のコードを確認する:

```bash
git diff origin/main...{不採用ブランチ名}
```

比較レポートと実際のコードから、以下の観点で取り込むべき改善点を特定する:

- **設計**: より良い抽象化、責務分離、インターフェース設計
- **エッジケース処理**: 選定候補が見落としているバリデーションやエラーハンドリング
- **パフォーマンス**: より効率的なアルゴリズムやクエリ
- **テスト**: より網羅的なテストケースやテストユーティリティ
- **可読性**: より明確な命名、コメント、コード構造

### Step 4: 改善計画をリストアップ

取り込む改善点を具体的にリストアップする。各項目について:
- どの不採用候補由来か
- 何を改善するか
- 選定候補のどこに適用するか

> **重要**: 選定候補のアーキテクチャや基本設計は維持すること。取り込むのは部分的な改善であり、設計の根本的な変更ではない。

### Step 5: 改善の適用

選定候補のブランチ上で改善を実施する。

### Step 6: 品質チェック

`AGENTS.md` の「よく使うコマンド」に記載された以下を実行:
- lint
- typecheck（あれば）
- test

エラーがあれば修正する。

### Step 7: コミット

改善内容を Conventional Commits 形式でコミットする。

```bash
git add -A
git commit -m "refactor({scope}): 候補比較からのブラッシュアップを適用"
git push origin {{SELECTED_BRANCH}}
```

### Step 8: 結果の返却

以下の形式で結果を返却すること:

```
## ブラッシュアップ結果

### 取り込んだ改善点

| # | 由来 | 改善内容 | 適用箇所 |
|---|------|---------|---------|
| 1 | candidate-X | ... | ... |
| 2 | candidate-Y | ... | ... |

### 取り込まなかった点（理由付き）

| # | 由来 | 内容 | 見送り理由 |
|---|------|------|----------|
| 1 | candidate-X | ... | ... |

### 品質チェック結果

- lint: pass/fail
- typecheck: pass/fail
- test: pass/fail
```
````

---

### Phase 4: 結果報告

ブラッシュアップ結果をユーザー（または呼び出し元スキル）に提示する。

**単体起動時**: 結果を表示して完了。PR作成やブランチ削除は行わない（ユーザーに委ねる）。

**他スキルからの参照時**: 呼び出し元スキルが結果を受けてPR作成・ブランチ削除を実行する。

---

## エッジケース

- **候補が2つ未満**: 比較・ブラッシュアップは不要。そのまま呼び出し元に制御を返す
- **全候補 failed**: 比較をスキップし、失敗報告を返す
- **成功候補が1つだけ**: 比較・ブラッシュアップをスキップし、その候補をそのまま返す
