# Self Review

現在のブランチの変更差分に対してセルフレビューを実施します。

## 使い方

```
/self-review              # 通常モード（code-reviewer + design-reviewer）
/self-review --agents N   # マルチエージェントモード（N個の視点で並列レビュー）
```

### オプション

| オプション | デフォルト | 説明 |
|-----------|-----------|------|
| `--agents N` | なし（通常モード） | N個のエージェントが並列でレビューし、結果を統合する |

---

## 通常モード（デフォルト）

`/self-review` を引数なしで呼び出した場合の動作。

### 手順

#### 1. 変更差分の取得

```bash
# 現在のブランチの変更を確認
git log origin/main..HEAD --oneline
git diff origin/main..HEAD
```

#### 2. コードレビューの実施

`code-reviewer` エージェントに委譲してレビューを実施:

- コーディング規約準拠
- 品質・セキュリティ
- テスト充足
- E2Eテスト（対象機能の場合）

#### 3. 設計レビューの実施

`design-reviewer` エージェントに委譲して設計レビューを実施:

- 依存方向の検証
- パッケージ境界の検証
- Feature-based 構造の検証
- プロジェクト固有の設計検証

#### 4. レビュー結果の報告

以下の形式で報告:

1. **問題点**: 修正が必要な箇所
2. **改善提案**: より良い実装の提案（任意）
3. **良い点**: 評価できる実装（任意）

#### 5. 問題がある場合

指摘事項を修正し、再度セルフレビューを実施。
修正完了後、変更をコミット。

---

## マルチエージェントモード（`--agents N`）

`/self-review --agents 3` のように `--agents` オプションを指定した場合の動作。

複数の独立したエージェントが異なる視点でレビューを並列実行し、結果を統合することで、単一エージェントでは見落としがちな問題を早期発見できます。

### いつ使うか

- PRマージ前の最終確認
- セキュリティ・パフォーマンスに敏感な変更
- 複数のコンポーネントにまたがる大規模変更
- 高精度なレビューが求められるクリティカルパスの変更

### 手順

#### 1. 変更差分の取得

```bash
git log origin/main..HEAD --oneline
git diff origin/main..HEAD
```

#### 2. エージェント数の確認

`--agents N` の N を確認する（例: `--agents 3` なら 3 エージェント）。
N は 2 以上 5 以下を推奨。

#### 3. N個のエージェントを並列でレビュー実行

各エージェントは独立した視点でレビューを担当する。N=3 の場合の例:

| エージェント | 担当視点 |
|------------|---------|
| エージェント1 | コード品質・規約・テスト充足（`code-reviewer` 相当） |
| エージェント2 | 設計・アーキテクチャ・依存関係（`design-reviewer` 相当） |
| エージェント3 | セキュリティ・パフォーマンス・エッジケース |

N=4 の場合は上記に加えて:

| エージェント | 担当視点 |
|------------|---------|
| エージェント4 | ドキュメント・コメント・API互換性 |

N=5 の場合はさらに:

| エージェント | 担当視点 |
|------------|---------|
| エージェント5 | 総合評価・トレードオフ分析 |

各エージェントは以下を独立して実施する:

```
1. 差分を分析
2. 担当視点でのレビューを実施
3. 指摘事項・改善提案・評価をリストアップ
```

#### 4. 結果の統合

全エージェントのレビュー結果を以下の形式で統合して報告:

```
## マルチエージェントレビュー結果（N エージェント）

### 重大度: Critical（必ず修正）
- [エージェントX] <指摘内容>

### 重大度: Warning（修正推奨）
- [エージェントX] <指摘内容>

### 改善提案（任意対応）
- [エージェントX] <提案内容>

### 良い点
- [エージェントX] <評価内容>

### 総評
<全エージェントの総合評価>
```

#### 5. 問題がある場合

Critical の指摘を優先して修正し、再度レビューを実施。
修正完了後、変更をコミット。

---

## 通常モードとマルチエージェントモードの使い分け

| 状況 | 推奨モード |
|------|-----------|
| 日常的な小規模変更 | 通常モード |
| WIPの途中確認 | 通常モード |
| PRマージ前の最終確認 | `--agents 3` |
| セキュリティに関わる変更 | `--agents 3` 以上 |
| 大規模リファクタリング | `--agents 4` 以上 |
| クリティカルパスの変更 | `--agents 5` |

## hooks による自動トリガー

`Write` / `Edit` ツール実行後、`auto-self-review.sh` フックが自動的に起動し、
変更ファイルの種別に応じてセルフレビューの実施を推奨します。

- 対象: `.sh`, `.json`, `.yaml`, `.md`, `.js`, `.ts`, `.py` など
- スキップ: 一時ファイル、`node_modules/`、ビルド成果物、画像・バイナリ、ロックファイル

フックはあくまで「推奨通知」を出力するもので、`/self-review` の呼び出しは手動です。
