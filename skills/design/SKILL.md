# 設計

要件チケット（またはドキュメント）をもとにコードベースを分析し、設計ドキュメントを作成します。

生成した設計ドキュメントは `/auto-develop` の Phase 2 としても自動実行されます。

---

## 入力パラメータ

親Issue番号またはパスとオプション: $ARGUMENTS

### パース方法

`$ARGUMENTS` を以下のルールで解釈する:

判定順序:

1. **フラグ処理**: `--auto`、`--candidates N`（または `-c N`）を検出・分離する
2. **Issue番号判定**: 先頭の `#` を除去した文字列が純粋な数値（`/^\d+$/`）なら親Issue番号として扱う（`#42` と `42` は同一）
3. **ファイルパス判定**: パス区切り（`/`）を含む、またはファイル拡張子（`.md` 等）を持つ場合はファイルパスとして扱う
4. **いずれにも該当しない場合**: エラーとしてユーザーに入力の確認を求める

- **`--auto`**: 自動モード。ユーザー確認なしで設計を完了する（auto-developから呼ばれる場合）
- **`--candidates N`**（または **`-c N`**）: N個の設計書を生成して比較・選定する（デフォルト: 1）。Nは2以上の整数を指定する
- 例:
  - `#42` → 親Issue #42 の要件を読み取り設計
  - `42` → 親Issue #42（`#` は省略可）
  - `docs/requirements/auth.md` → 要件ドキュメントから設計
  - `#42 --auto` → 自動モードで設計（auto-developからの呼び出し）
  - `#42 -c 3` → 3案の設計書を生成し、比較・選定
  - `#42 -c 3 --auto` → 自動モードで3案を比較し、最適案を自動選定

---

## 手順

### Step 1: プロジェクト理解

`CLAUDE.md` を読み、以下を把握:

- 技術スタック
- ディレクトリ構成・パッケージ構造
- 開発原則・設計規約
- ドキュメントマップ（既存の設計ドキュメント）

### Step 2: 要件の読み込み

入力に応じて要件を取得する:

**親Issue番号の場合:**
```bash
gh issue view {番号} --json title,body,state,labels,number
```
Issue本文から要件を把握する。要件ドキュメントへのリンクがあればそのファイルも読み込む。

**ファイルパスの場合:**
- ファイルの場合: そのファイルを読み込む
- ディレクトリの場合: 配下の全 `.md` ファイルを読み込む

### Step 3: コードベース分析

要件に関連する既存コードを分析する:

1. **関連モジュールの特定**: 要件が影響するモジュール・ディレクトリを Glob/Grep で探索
2. **既存パターンの把握**: 類似機能の実装パターンを確認（同じ層の既存コードを参照）
3. **依存関係の調査**: 変更が影響する上流・下流のモジュールを特定
4. **技術的制約の確認**: フレームワークの制約、既存APIとの整合性

### Step 4: 設計ドキュメント作成

テンプレートを読み込む:

1. `CLAUDE.md` に設計ドキュメントテンプレートのパスが指定されている場合はそれを使用
2. プロジェクトルートに `docs/templates/design-doc.md` が存在する場合はそれを使用
3. いずれもない場合は `skills/design/templates/design-doc.md` を使用

`{slug}` は以下の手順で生成:
1. 親Issueのタイトルから機能名を取得する
2. 英数字・ハイフン以外の文字（日本語含む）はそのままローマ字変換せず、スペース・アンダースコアをハイフンに変換
3. 大文字を小文字に変換し、連続するハイフンを1つにまとめ、先頭・末尾のハイフンを削除
4. 既存ファイルと重複する場合は末尾に `-2`, `-3` 等の連番を付与
5. 例: "User Authentication Feature" → `user-authentication-feature`

#### candidates == 1 の場合（デフォルト）

テンプレートに沿って設計ドキュメントを1つ作成する:

- **保存先**: `docs/design/{slug}.md`
  - `CLAUDE.md` に設計ドキュメントの保存先が指定されている場合はそれに従う
- **内容**:
  - アーキテクチャ決定: 実装方針とその理由、変更対象モジュール、既存コードとの関係
  - IF/API: モジュール間の境界となるインターフェース契約（該当する場合）
  - データモデル: DBスキーマ変更、データ構造（該当する場合）
  - 実装計画: チケット分解案、依存関係、リスク
  - 該当しないセクションはセクションごと削除する

→ Step 5 へ進む。

#### candidates > 1 の場合

N個の設計ドキュメントを生成する。各案は **異なるアーキテクチャ方針** で作成すること（例: パフォーマンス重視 vs 保守性重視、既存パターン踏襲 vs リファクタリング併用など）。

- **保存先**: `docs/design/{slug}-candidate-{i}.md`（i = 1, 2, ..., N）
  - `CLAUDE.md` に設計ドキュメントの保存先が指定されている場合はそのディレクトリ内に保存
- **内容**: テンプレートに沿って作成し、加えて各案の冒頭に以下のセクションを追加する:
  - `## アプローチ概要` — この案のアーキテクチャ方針と他の案との違いを簡潔に説明
  - それ以降は通常の設計ドキュメントと同様

→ Step 4b へ進む。

### Step 4b: 比較・選定（candidates > 1 の場合のみ）

N個の設計書を以下の観点で比較評価する:

| 評価観点 | 説明 |
|---------|------|
| 実現可能性 | 技術的な実現難易度、チームのスキルセットとの適合性 |
| 変更の最小性 | 既存コードへの影響範囲の小ささ |
| 保守性 | 長期的なメンテナンスのしやすさ |
| リスク | 実装・運用上のリスクの低さ |

**対話モード（`--auto` なし）:**

比較レポートを以下の形式でユーザーに提示し、どの案を採用するか選定を求める:

```
## 設計案 比較レポート

| 観点 | 案1: {方針名} | 案2: {方針名} | ... |
|------|-------------|-------------|-----|
| 実現可能性 | {評価} | {評価} | ... |
| 変更の最小性 | {評価} | {評価} | ... |
| 保守性 | {評価} | {評価} | ... |
| リスク | {評価} | {評価} | ... |

### 推奨: 案{X}
{推奨理由}

どの案を採用しますか？（番号で回答）
```

ユーザーが案を選定したら、選定された案を `docs/design/{slug}.md` として保存し、候補ファイル（`{slug}-candidate-{i}.md`）をすべて削除する。

**自動モード（`--auto`）:**

比較評価に基づき、最も推奨される案を自動選定する。選定された案を `docs/design/{slug}.md` として保存し、候補ファイルをすべて削除する。

→ Step 5 へ進む。

### Step 5: ユーザー確認（通常モード）

`--auto` が **指定されていない** 場合:

設計ドキュメントの内容をユーザーに提示し、フィードバックを求める。指摘があれば修正する。

`--auto` の場合はこのステップをスキップする。

> **注**: candidates > 1 かつ対話モードの場合、Step 4b で既にユーザーと対話して案を選定済みのため、ここでは選定後の最終設計ドキュメントに対するフィードバックを求める。

### Step 6: 結果の返却

**通常モード（対話）:**

```
## 設計完了

- 設計ドキュメント: `docs/design/{slug}.md`
- 親チケット: #{番号}

実装チケットの作成に進むには:
/auto-develop --from-issue #{番号}
```

**自動モード（`--auto`、candidates == 1）:**

以下の JSON 形式のみを最終出力として返却する:

```json
{
  "parent": 親チケット番号,
  "design_doc": "docs/design/{slug}.md"
}
```

**自動モード（`--auto`、candidates > 1）:**

以下の JSON 形式のみを最終出力として返却する:

```json
{
  "parent": 親チケット番号,
  "design_doc": "docs/design/{slug}.md",
  "candidates_evaluated": N
}
```
