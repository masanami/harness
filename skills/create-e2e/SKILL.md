# 対話型E2Eテスト実装

ユーザーの動作確認を組み込んだ対話型E2Eテスト実装スキルです。正常系はユーザーが実際に動作確認した結果をテストケース化することで、E2Eテスト自体の正しさを保証します。

## 設計方針

- **正常系**: ユーザーが実際に動作確認 → 確認済みの動作をテストコード化（テストのオラクルがユーザー確認）
- **異常系・エッジケース**: 正常系が確認できていれば、静的レビューで十分

---

## 入力パラメータ

対象機能の指定: $ARGUMENTS

| 入力形式 | 解釈 |
|---------|------|
| Issue番号（例: `#42`） | 指定Issueの機能に対するE2Eテスト |
| PR番号（例: `PR #10`） | 指定PRの変更に対するE2Eテスト |
| 機能名・パス | 指定された機能に対するE2Eテスト |
| なし | ユーザーに対象を確認 |

### オプション

- **`--auto`**: 自動モード。Phase 2（承認）・Phase 3（動作確認）をスキップし、全テストケースを静的に実装する

| モード | Phase 2 | Phase 3（正常系） | Phase 4（異常系） |
|--------|---------|-----------------|-----------------|
| デフォルト | ユーザー承認 | 動作確認付き実装 | 静的実装 |
| `--auto` | スキップ | 静的実装 | 静的実装 |

---

## Phase 1: テスト設計

### Step 1-1: プロジェクトコンテキストの確認

プロジェクトルートの `CLAUDE.md` を読み、以下を把握する:
- テスト方針（E2Eテストのガイドライン）
- ドキュメントマップ（E2Eテスト設計書の場所）
- よく使うコマンド（E2Eテスト実行コマンド）

### Step 1-2: 対象機能の分析

入力パラメータに基づき、対象機能の要件・仕様を把握する:
- Issue/PRの場合: `gh issue view` / `gh pr view` で内容を取得
- 機能名の場合: 関連コードを調査
- 設計書・要件書がある場合はそれも参照

### Step 1-3: 既存E2Eテストのパターン調査

既存のE2Eテストコードを調査し、プロジェクトのパターンを把握する:
- テストフレームワーク（Playwright, Cypress 等）
- ディレクトリ構成・ファイル命名規則
- 共通ユーティリティ・ヘルパー関数
- Page Object等の抽象化パターン（使用されている場合）
- 認証ヘルパー・テストデータのセットアップ方法

### Step 1-4: テストケース一覧の設計

テストケースを以下の3分類で設計する:

| 分類 | 説明 | 動作確認 |
|------|------|---------|
| **正常系** | ビジネス上重要なハッピーパス | ユーザーが動作確認 |
| **異常系** | エラーハンドリング・バリデーション | 静的実装 |
| **エッジケース** | 境界値・特殊条件 | 静的実装 |

---

## Phase 2: ユーザー承認

> **`--auto` の場合**: このフェーズをスキップし、Phase 3 へ進む。

テスト計画を以下の形式でユーザーに提示する:

```
## E2Eテスト計画

### 対象機能
- {機能名・概要}

### 正常系（動作確認付き）
1. {テストケース名}: {シナリオ概要}
2. ...

### 異常系
1. {テストケース名}: {シナリオ概要}
2. ...

### エッジケース
1. {テストケース名}: {シナリオ概要}
2. ...

この計画で進めてよいですか？修正があれば指示してください。
```

ユーザーの修正指示があれば反映し、承認を得てから次に進む。

---

## Phase 3: 正常系の実装

> **`--auto` の場合**: 動作確認をスキップし、Phase 4 と同様に全正常系テストケースを静的に一括実装する。以下の対話フローは実行しない。

正常系テストケースごとに以下のサイクルを繰り返す。

### 各ケースのフロー

#### Step 3-1: データ準備

テストのsetup相当の処理を実行し、ユーザーが検証できる状態を作る:
- 必要なテストデータの投入（seed、API呼び出し等）
- アプリケーションの起動確認
- テスト対象画面・APIへのアクセス手順を提示

ユーザーに以下を伝える:

```
## 正常系 {N}/{総数}: {テストケース名}

### 準備完了
{セットアップ内容の説明}

### 動作確認手順
1. {具体的な操作手順}
2. {期待される結果}

上記の手順で動作確認をお願いします。結果を教えてください（OK / NG / 修正点）。
```

#### Step 3-2: ユーザー動作確認

ユーザーの回答を待つ:

| ユーザー回答 | アクション |
|------------|---------|
| **OK** | Step 3-3 に進む |
| **NG + 詳細** | 問題を分析し、修正案を提示。修正後、再度 Step 3-1 から |
| **修正点あり** | テストシナリオを修正し、再度 Step 3-1 から |

#### Step 3-3: テストケース化

ユーザーがOKと判断したシナリオをE2Eテストコードに落とす:
- プロジェクトの既存パターンに従う
- 共通ユーティリティ・ヘルパーを再利用
- `data-testid` 属性が必要な場合は先にコンポーネントに追加

実装したテストコードをユーザーに簡潔に報告し、次のケースへ進む。

---

## Phase 4: 異常系・エッジケースの実装

正常系の動作確認がすべて完了したら、異常系・エッジケースを静的に実装する。

- ユーザーの動作確認は不要（静的レビューで十分）
- 正常系で確立したパターンに従い一括実装
- 実装完了後、追加したテストケースの一覧をユーザーに報告

```
## 異常系・エッジケース実装完了

### 追加したテスト
| 分類 | テストケース名 | 検証内容 |
|------|-------------|---------|
| 異常系 | ... | ... |
| エッジケース | ... | ... |
```

---

## Phase 5: 全テスト実行・レポート

### Step 5-1: 全E2Eテスト実行

`CLAUDE.md` の「よく使うコマンド」に記載されたE2Eテスト実行コマンドを使用して、全テストを実行する。`CLAUDE.md` が存在しない場合は、先に `/init-project` を実行してプロジェクト設定を生成すること。

### Step 5-2: 結果レポート

```
## E2Eテスト実装結果

### サマリー
- 正常系（動作確認済み）: N件
- 異常系: N件
- エッジケース: N件
- 合計: N件

### テスト実行結果
- パス: N件
- 失敗: N件

### 失敗テスト（ある場合）
1. テスト名: エラー内容と対応方針

### 作成・変更したファイル
- {ファイルパス}: {概要}
```

失敗テストがある場合は修正を試み、再実行する。

---

## 注意事項

- コンポーネントに `data-testid` 属性が必要な場合は、先に追加する
- メール送信を伴うテストは、送信トリガーまでの検証に留める
- 正常系の動作確認では、ユーザーが実際にアプリを操作できる状態を必ず整える
- テストコードはプロジェクトの既存パターンに厳密に従う
