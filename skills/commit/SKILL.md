# Conventional Commit

変更をConventional Commits形式でコミットします。コミット前にセルフレビューと品質チェックを実施します。

## 手順

### 1. セルフレビュー

`/self-review` を実行し、`code-reviewer` と `design-reviewer` エージェントに委譲してレビューを実施:

- コーディング規約準拠
- 品質・セキュリティ
- テスト充足
- 設計の妥当性（依存方向、パッケージ境界等）

#### 問題がある場合

指摘事項を修正し、再度セルフレビューを実施してすべての問題が解消されるまで繰り返す。

### 2. コードの簡潔化（推奨）

`/simplify`（code-simplifier エージェントに委譲）を実行してコードを簡潔化:

- 冗長なコードの除去
- 可読性の向上
- 不要な複雑性の排除

> **注意**: このステップはオプションですが実施を推奨します。simplifier がコードを変更した場合は次のステップの品質チェックが**必須**になります。

### 3. 品質チェック（simplifier が変更を加えた場合は必須、それ以外も推奨）

`/quality-check` を実行し、lint・型チェック・テストを一括で確認:

- リント: コーディング規約の自動チェック
- 型チェック: 型安全性の確認（TypeScript等）
- テスト: 既存テストがすべて通過することを確認

失敗した場合はエラーを修正し、再度品質チェックを実施。

### 4. 変更の確認

```bash
git status
git diff --staged
git diff
```

ステージされていない変更がある場合は、関連する変更をステージングします。

### 5. 変更の分析

変更内容を分析し、適切なコミットタイプを決定:

| Type       | 説明                      |
|------------|-------------------------|
| `feat`     | 新機能追加                   |
| `fix`      | バグ修正                    |
| `docs`     | ドキュメントのみの変更             |
| `style`    | コードの意味に影響しない変更（フォーマット等） |
| `refactor` | バグ修正でも機能追加でもないコード変更     |
| `perf`     | パフォーマンス改善               |
| `test`     | テストの追加・修正               |
| `chore`    | ビルドプロセスやツールの変更          |
| `ci`       | CI設定の変更                 |

### 6. コミットメッセージの作成

フォーマット:
```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

ルール:
- scopeはプロジェクトの規約に従う（パッケージ名、機能領域等）
- descriptionは日本語で記述、命令形（「〜を追加」「〜を修正」）、50文字以内目安
- bodyには変更の詳細を記述（必要な場合）
- footerには関連Issue番号を記載（`Refs: #123`）

### 7. コミット実行

```bash
git commit -m "<message>"
```

### 8. コミット粒度

- 1つのコミットは1つの論理的な変更
- 動作する状態でコミット
- 大きな変更は複数のコミットに分割
