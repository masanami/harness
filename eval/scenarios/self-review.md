# self-review 評価シナリオ

## スキル概要

`/self-review` は現在のブランチの変更差分に対してセルフレビューを実施するスキルです。
`code-reviewer` および `design-reviewer` エージェントに委譲して、コーディング規約・品質・設計の観点でレビューを行います。code-reviewer は「問題点・改善提案・良い点」の形式で、design-reviewer は「依存違反・境界侵犯・構造問題・推奨対応」の形式でそれぞれ報告します。

**参照**: [SKILL.md](../../skills/self-review/SKILL.md)

---

## 評価観点

- **指示への追従率**: `git diff origin/main..HEAD` を取得し、code-reviewer と design-reviewer の両方に委譲しているか
- **生成物の正確さ**: code-reviewer が「問題点・改善提案・良い点」、design-reviewer が「依存違反・境界侵犯・構造問題・推奨対応」の形式でそれぞれ報告されているか
- **副作用の有無**: レビュー実行中にファイルの変更・コミット・プッシュが発生していないか
- **ユーザーへの確認タイミング**: 問題点がある場合に修正を促し、修正完了後にコミットを示唆しているか

## テストケース

### TC-01: コーディング規約違反を含む変更差分のレビュー

- **前提**:
  - main ブランチから `feature/test-self-review` ブランチを作成済み
  - CLAUDE.md にコーディング規約（例: インデントはスペース2つ、関数名はcamelCase等）が定義されている
  - コーディング規約に違反するコード変更（例: タブインデント、snake_case関数名）をコミット済みだが未プッシュ

- **入力**:
  ```text
  /self-review
  ```

- **期待する動作**:
  1. `git log origin/main..HEAD --oneline` および `git diff origin/main..HEAD` を実行して差分を取得する
  2. code-reviewer エージェントにコーディング規約違反を含む差分を渡してレビューを依頼する
  3. design-reviewer エージェントに設計観点のレビューを依頼する
  4. 「問題点・改善提案・良い点」の形式でレビュー結果を報告する
  5. コーディング規約違反を「問題点」として指摘する
  6. 修正と再レビューを促す

- **合格基準**:
  - [ ] `git diff origin/main..HEAD` が実行されている
  - [ ] code-reviewer への委譲が実施されている
  - [ ] design-reviewer への委譲が実施されている
  - [ ] code-reviewer の報告にコーディング規約違反が「問題点」として明示されている
  - [ ] レビュー後にファイルが変更されていない（`git status` でクリーン）

- **チェック方法**:
  ```bash
  # レビュー実行前後でファイルが変更されていないことを確認
  git status
  # 出力: "nothing to commit, working tree clean" であること

  # レビュー報告に問題点セクションが含まれることを目視確認
  # （AIの会話ログを確認）
  ```

### TC-02: 問題のない変更差分のレビュー

- **前提**:
  - main ブランチから `feature/test-self-review-clean` ブランチを作成済み
  - コーディング規約に準拠し、テストが充足した適切なコード変更をコミット済み

- **入力**:
  ```text
  /self-review
  ```

- **期待する動作**:
  1. 差分を取得してレビューを実施する
  2. 問題点がない場合は「問題点: なし」と明記する
  3. 良い点があれば記載する
  4. 次のステップ（PRの作成等）を案内する

- **合格基準**:
  - [ ] 問題が検出されなかった旨が明示されている（code-reviewer: 「問題点: なし」等、design-reviewer: 「依存違反: なし」等）
  - [ ] 問題のないコードに対して誤った指摘（誤検知）をしていない
  - [ ] 次のアクションが示されている

- **チェック方法**:
  ```bash
  # レビュー実行後もブランチ・ファイルが変更されていないことを確認
  git status
  git branch --show-current
  ```

### TC-03: 差分が空の場合のレビュー（境界条件）

- **前提**:
  - main ブランチ上で作業しているか、feature ブランチを切ったが変更がない状態

- **入力**:
  ```text
  /self-review
  ```

- **期待する動作**:
  1. 差分を取得しようとする
  2. 差分が空であることを検知してユーザーに通知する
  3. 「レビュー対象の変更がありません」等のメッセージを返す
  4. 異常終了しない

- **合格基準**:
  - [ ] エラーなく正常終了している（例外・クラッシュがない）
  - [ ] 差分がないことをユーザーに伝えている
  - [ ] 空の差分でレビューを「PASS」と誤報告していない

- **チェック方法**:
  ```bash
  # 差分が空であることを確認
  git diff origin/main..HEAD
  # 出力が空であること
  ```

### TC-04: セキュリティ上の問題を含む変更のレビュー

- **前提**:
  - main ブランチから feature ブランチを作成済み
  - 平文パスワード・APIキーのハードコーディング等、セキュリティ上の問題を含むコードをコミット済み

- **入力**:
  ```text
  /self-review
  ```

- **期待する動作**:
  1. 差分を取得してレビューを実施する
  2. セキュリティリスクを「問題点」として明確に指摘する
  3. 修正方法（環境変数の利用等）を「改善提案」として提示する
  4. 修正を促す

- **合格基準**:
  - [ ] セキュリティリスクが「問題点」として指摘されている
  - [ ] 指摘が具体的で、対象のファイル・行が特定されている
  - [ ] 改善方法が提案されている

- **チェック方法**:
  - 会話ログを確認し、セキュリティリスクへの言及があるかを目視確認
  - 指摘が曖昧でなく、具体的であることを確認

### TC-05: レビュー後の副作用確認

- **前提**:
  - 変更を含む feature ブランチが存在する（TC-01 の環境を再利用可能）

- **入力**:
  ```text
  /self-review
  ```

- **期待する動作**:
  - レビューが完了するまでの間、ファイルの作成・編集・削除が発生しない
  - コミット・プッシュが発生しない
  - ブランチの切り替えが発生しない

- **合格基準**:
  - [ ] `git status` がレビュー前後で同じ状態を示す
  - [ ] `git log` にレビュー中に作成された余分なコミットが存在しない
  - [ ] `git branch` でブランチが変更されていない

- **チェック方法**:
  ```bash
  # レビュー実行前に記録
  git log --oneline -5 > /tmp/before-review.txt
  git status >> /tmp/before-review.txt

  # /self-review 実行

  # レビュー実行後に比較
  git log --oneline -5 > /tmp/after-review.txt
  git status >> /tmp/after-review.txt
  diff /tmp/before-review.txt /tmp/after-review.txt
  # 差分がないこと
  ```
