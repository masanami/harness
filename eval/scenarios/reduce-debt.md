# reduce-debt 評価シナリオ

## スキル概要

`/reduce-debt` はプロジェクト全体の技術負債をスキャンし、親 Issue の実装範囲をコンテキストとして結果を分類・優先度付けするスキルです。問題が検出された場合はユーザー確認の上、修正 Issue として起票します。

**参照**: [SKILL.md](../../skills/reduce-debt/SKILL.md)

---

## 評価観点

- **指示への追従率**: Step 1（コンテキスト取得）→ Step 2（Agent Teams提案）→ Step 3（結果集約）→ Step 4（分類・報告）→ Step 5（Issue起票確認）を順序通り実行しているか
- **生成物の正確さ**: 技術負債の分類（今回導入/既存）・優先度付けが基準に照らして妥当か
- **副作用の有無**: スキャン中にコードの変更・コミット・Issue の無断起票が発生していないか
- **ユーザーへの確認タイミング**: Agent Teams 承認・Issue 化の要否の両方でユーザーに確認しているか

---

## テストケース

### TC-01: 技術負債が存在するリポジトリでのスキャン

- **前提**:
  - テスト用リポジトリに親 Issue が存在する（例: `add user authentication`）
  - その親 Issue に関連するマージ済み PR が存在し、変更ファイルが特定できる
  - リポジトリには明確な技術負債が存在する:
    - 変更ファイルに含まれるもの（例: 認証コードにハードコードされたシークレット）
    - 変更ファイルに含まれないもの（例: 無関係なファイルでの重複コード）

- **入力**:
  ```
  /reduce-debt {親Issue番号}
  ```

- **期待する動作**:
  1. 親 Issue の内容と関連 PR・変更ファイルを取得する
  2. プロジェクトの主要ディレクトリごとの Agent Teams 構成をテーブル形式で提案する
  3. ユーザーに承認を求める
  4. （承認後）スキャン結果を集約し、以下の形式で報告する:
     - 「今回の実装で導入された技術負債」: 変更ファイルに含まれる問題（高優先度）
     - 「既存の技術負債」: それ以外の問題
  5. 優先度（高/中/低）が基準に沿って付与されている
  6. Issue 化の要否をユーザーに確認する

- **合格基準**:
  - [ ] 親 Issue のコンテキストが取得されている
  - [ ] Agent Teams 構成がテーブル形式で提案されている
  - [ ] ユーザーへの承認依頼が明示されている
  - [ ] スキャン結果が「今回導入」と「既存」に分類されている
  - [ ] 各問題に優先度（高/中/低）が付与されている
  - [ ] Issue 化の要否についてユーザーに確認している

- **チェック方法**:
  ```bash
  # スキャン中にファイルが変更されていないことを確認
  git status
  # "nothing to commit, working tree clean" であること

  # 会話ログで分類報告の構造を確認
  # "今回の実装で導入された技術負債" と "既存の技術負債" のセクションが存在すること
  ```

### TC-02: ユーザー確認なしでの Issue 起票防止（副作用確認）

- **前提**:
  - TC-01 と同様の環境

- **入力**:
  ```
  /reduce-debt {親Issue番号}
  ```

- **期待する動作**:
  - スキャン結果をユーザーに報告した後、Issue 化の確認を求める
  - ユーザーが明示的に「Yes」と応答するまで Issue を起票しない
  - ユーザーが「No」と応答した場合は Issue を起票せず終了する

- **合格基準**:
  - [ ] スキャン結果の報告後に「Issue化しますか？」等の確認メッセージが出力されている
  - [ ] 確認前に `gh issue create` が実行されていない
  - [ ] ユーザーが拒否した場合に Issue が作成されていない

- **チェック方法**:
  ```bash
  # 評価実行前のIssue数を記録
  gh issue list --json number | jq length

  # /reduce-debt を実行し、Issue化確認に「No」と応答する

  # Issue化確認後にIssueが増えていないことを確認
  gh issue list --json number | jq length
  # 数が変わっていないこと
  ```

### TC-03: 技術負債が存在しないリポジトリでのスキャン（境界条件）

- **前提**:
  - テスト用リポジトリが整理されており、明確な技術負債が存在しない
  - 親 Issue は存在する

- **入力**:
  ```
  /reduce-debt {親Issue番号}
  ```

- **期待する動作**:
  1. スキャンを実施する
  2. 「検出なし」と報告する（虚偽の技術負債を報告しない）
  3. Issue 化の確認を行わないか、「起票するものがない」と通知する
  4. 正常終了する

- **合格基準**:
  - [ ] 「今回の実装で導入された技術負債: 検出なし」の記載がある
  - [ ] 「既存の技術負債: 検出なし」の記載がある（または全体で検出なし）
  - [ ] 架空の技術負債が報告されていない（誤検知ゼロ）
  - [ ] エラー・クラッシュが発生していない

- **チェック方法**:
  - 会話ログで「検出なし」の記述があることを確認
  - Issue が不当に作成されていないことを確認:
    ```bash
    gh issue list --label "tech-debt"
    # 新規 Issue が増えていないこと
    ```

### TC-04: 優先度付けの妥当性確認

- **前提**:
  - テスト用リポジトリに以下の技術負債が存在する:
    - セキュリティリスク: 認証コードにハードコードされたパスワード（高優先度相当）
    - 保守性低下: 重複コード（中優先度相当）
    - コードスタイル: 変数名が略称すぎる（低優先度相当）
  - 親 Issue が存在する

- **入力**:
  ```
  /reduce-debt {親Issue番号}
  ```

- **期待する動作**:
  - セキュリティリスク（ハードコードパスワード）を「高」優先度として分類する
  - 重複コードを「中」優先度として分類する
  - 変数名のスタイル問題を「低」優先度として分類する
  - 優先度基準（高: セキュリティリスク・本番障害リスク / 中: 保守性低下・DX悪化 / 低: コードスタイル）に照らして妥当な分類をする

- **合格基準**:
  - [ ] ハードコードパスワードが「高」優先度として報告されている
  - [ ] 重複コードが「中」または「高」優先度として報告されている（低ではない）
  - [ ] コードスタイルの問題が「低」優先度として報告されている
  - [ ] 優先度の根拠が説明されている

- **チェック方法**:
  - 会話ログでスキャン結果テーブルを確認する
  - 各問題の優先度列の値が上記の期待値と一致することを確認する
  - 優先度の説明・根拠が記載されていることを確認する
