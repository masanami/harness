# AI駆動開発戦略

## 1. 概要

本ドキュメントは、AI駆動開発における**チーム体制・オーナーシップモデル・品質保証フロー**を定義する。

核となる考え方:

- **開発者はレビュー・承認に集中し、AIエージェントが実装を担う**
- **オーナーシップにより承認待ちのブロッキングを排除する**
- **フェーズに応じた品質保証レベルで、人的レビュー負荷と品質のバランスを取る**

### 2.2 オーナーシップモデル

AI駆動開発において最も重要な設計判断は**オーナーシップの付与**である。

各開発者が特定領域の責任者として、設計からマージまで一貫して担当する。これにより「他者のApprove待ち」というブロッキングを排除し、開発スループットを最大化する。

| 責任 | 担当 |
|------|------|
| 担当領域のアーキテクチャ決定 | オーナー開発者 |
| AIエージェントへのタスク指示・コンテキスト提供 | オーナー開発者 |
| AI生成コードのレビュー・承認 | オーナー開発者 |
| PRマージ | オーナー開発者 |
| 担当領域のコード品質 | オーナー開発者 |
| 担当領域の技術ドキュメント管理 | オーナー開発者 |

> **ポイント**: オーナーシップは「自分の判断でマージまで完結できる」ことを意味する。ただし品質保証レベルに応じて、他開発者へのレビュー依頼は行う（セクション5参照）。

## 3. 役割定義

### 3.1 PdM / PdO

- 機能要件の詳細化・優先順位決定
- プロダクトロードマップの策定・管理
- 開発者間の調整・ブロッカー解消
- リリース判断
- 基本設計・概要設計など広範に影響のあるドキュメントの品質管理

**タスク管理**:
- 機能要件を親チケットとして作成
- 個別の実装タスクは開発者に一任
- 基盤機能（DB設計、セキュリティ、外部連携等）に関わるチケットのみ事前レビュー

### 3.2 開発者

各開発者は特定領域のオーナーとして、AIエージェントチームを統率する。

- AIエージェントへのタスク指示・コンテキスト提供
- AI生成コードのレビュー・承認
- 担当サービスのPRマージ権限を保持
- アーキテクチャ決定・技術選定
- 他開発者との技術的な調整
- 詳細設計・API仕様など担当サービスの技術ドキュメント管理

### 3.3 QAエンジニア

- 開発フェーズ完了後の機能全体を通したQA（動作確認・E2Eテスト）
- テスト戦略の策定・テスト観点の整備

### 3.4 AIエージェント

**1エージェントが1チケットを最初から最後まで完結させる**構成をとる。

```
開発者
├── エージェント 1 ──→ チケット A（実装→テスト→セルフレビュー→PR作成）
├── エージェント 2 ──→ チケット B（実装→テスト→セルフレビュー→PR作成）
├── エージェント 3 ──→ チケット C（実装→テスト→セルフレビュー→PR作成）
└── エージェント 4 ──→ チケット D（実装→テスト→セルフレビュー→PR作成）
```

Claude Code を使用する場合:

| 階層 | 役割 | 例 |
|------|------|-----|
| メインセッション | 1チケットの開発サイクル全体を管理 | `/para-impl 123` で自律実行 |
| サブエージェント | メインセッション内で特定タスクを実行 | コードベース探索、テスト実行 |
| Skills | 定型タスクの自動実行 | `/commit`, `/self-review`, `/test` |

## 5. 品質保証とレビュー負荷の設計

### 5.1 基本方針

AI駆動開発では「AIが実装し、人間がレビュー・承認する」形をとる。ここで課題となるのが**人的レビュー負荷**である。

現在のAIエージェントはコード品質が十分に高く、コードそのものに問題があるケースは少ない。一方で、**設計方針の誤りは大きな手戻りを生む**。そこで本戦略では**設計レビューをコードレビューより優先**し、人的リソースを「正しい方向に向かっているか」の判断に集中させる。

コードの品質担保は**テスト（特にE2Eテスト）を主軸**とし、プロダクトのフェーズに応じて品質保証レベルを設定、人的レビューとテストの範囲・深度をコントロールする。

> 品質保証レベル・品質ゲートの詳細定義は [品質ゲート定義](./workflows/quality-gates.md) を参照。

### 5.2 フェーズ別のレビュー体制

レビューを**設計レビュー**と**コードレビュー**の2つの観点に分けて運用する。**設計レビューを優先**し、人的リソースは「方向性が正しいか」の判断に集中させる。

| 観点 | タイミング | 対象 | 目的 | 優先度 |
|------|-----------|------|------|--------|
| 設計レビュー | 実装前・PR作成後 | 設計方針・影響範囲・実行計画との整合 | 手戻りの防止 | **高** |
| コードレビュー | PR作成後 | 実装コード・テスト | 品質の担保 | 低 |

設計レビューは実装前だけでなく、PR作成後にも実施する。実装を通じて初めて見える設計上の課題や、コンフリクト解消による設計意図の逸脱を検出するためである。

> **原則**: AIのコード品質は十分高い。人間が時間をかけるべきは「何を作るか・どう作るか」の設計判断であり、コードの精読ではない。設計が正しければ、コードの問題は自動テストとセルフレビューで十分に検出できる。

```
Lv.0  スパイク・技術検証          レビュー負荷: なし
------------------------------------------------------------

  エージェント/開発者
  +----------+
  | 実装     |
  | テスト   |-----> マージ（又は使い捨て）
  | セルフ   |
  | レビュー |
  +----------+

  設計レビュー : なし
  コードレビュー: なし

  * 技術検証・スパイク・使い捨てプロトタイプが対象
  * 個人の判断で即マージ可能
  * 本番コードへの流用時は適切なLv.で再レビュー


Lv.1  プロトタイプ・PoC           レビュー負荷: 低
------------------------------------------------------------

  エージェント              開発者
  +----------+           +--------------+
  | 実装     |           | 動作確認     |
  | テスト   |---PR----->| E2Eテスト    |---> マージ
  | セルフ   |           +--------------+
  | レビュー |
  +----------+

  設計レビュー : 重要箇所で実施（実装後に設計適合の評価を含む）
  コードレビュー: なし

  * 素早い仮説検証を優先
  * 重要箇所は設計方針の確認と、実装後の設計適合評価を実施
  * コードレビューは行わない（AIのセルフレビュー+自動テストで担保）


Lv.2  ベータ版・PMF検証           レビュー負荷: 中
------------------------------------------------------------

  【設計レビュー】
  エージェント <---> 開発者
  +------------------+    +------------------+
  | 設計方針を提案   |--->| 重要機能の       |
  | 影響範囲を特定   |<---| 設計方針を確認   |
  | 実行計画を提示   |    | 承認             |
  +------------------+    +------------------+
                               |
                               v 承認後
  【コードレビュー】
  エージェント          担当開発者    他開発者
  +----------+       +----------+  +----------+
  | 実装     |       | コード   |  | 必要に   |
  | テスト   |--PR-->| レビュー |->| 応じて   |-> マージ
  | セルフ   |       | 動作確認 |  | レビュー |
  | レビュー |       +----------+  +----------+
  +----------+

  設計レビュー : 原則すべての変更で実施
  コードレビュー: クリティカル箇所のみ

  * 原則すべての変更で設計レビューを実施し、方向性を確認した上で実装に入る
  * コードレビューはセキュリティ・認証等のクリティカル箇所に限定
  * 担当開発者のApproveで原則マージ可能
  * 複雑な機能・クロスドメインは設計レビュー段階で他開発者にも相談


Lv.3  製品版                      レビュー負荷: 高
------------------------------------------------------------

  【設計レビュー】
  エージェント <---> 開発者 + 他開発者/PdM
  +------------------+    +--------------------+
  | 設計方針を提案   |--->| 設計方針を確認     |
  | 影響範囲を特定   |<---| 影響範囲の妥当性   |
  | 実行計画を提示   |    | 承認               |
  +------------------+    +--------------------+
                               |
                               v 承認後
  【コードレビュー】
  エージェント    担当開発者    他開発者      リリース前
  +----------+  +----------+  +----------+  +----------+
  | 実装     |  | コード   |  | 必要に   |  | 第三者   |
  | テスト   |->| レビュー |->| 応じて   |->| レビュー |-> マージ
  | セルフ   |  | 動作確認 |  | レビュー |  | (重要)   |
  | レビュー |  +----------+  +----------+  +----------+
  +----------+

  設計レビュー : 原則すべての変更で実施（他開発者/PdMを含む）
  コードレビュー: 重要箇所を中心に実施

  * 設計レビューで方向性・影響範囲を複数人で確認してから実装に入る
  * コードレビューは重要箇所を中心に実施（全行精読は不要）
  * リリース前に重要機能の第三者レビューを必須化
  * 規制対応がある場合は検証者の独立性を担保
```

#### レビュー・テスト早見表

| | 設計レビュー | コードレビュー | E2E | マージ権限 |
|------|------------|--------------|-----|-----------|
| Lv.0 | なし | なし | なし | 本人 |
| Lv.1 | 重要箇所で実施 | なし | 手動で動作確認 | オーナー |
| Lv.2 | 原則全件 | クリティカル箇所のみ | 主要フロー自動化 | オーナー |
| Lv.3 | 原則全件 | 重要箇所を中心に | 全フロー自動化+手動QA | オーナー + 第三者 |

> **読み方**: 上位レベルほど設計レビューとE2Eテストの範囲が先に広がり、コードレビューは限定的に留まる。人的リソースの投下先は「設計レビュー > E2Eテスト整備 > コードレビュー」の優先順位。

### 5.3 フェーズ別のテスト戦略

コードレビューを最小限に抑える代わりに、**テストを品質担保の主軸**とする。特にE2Eテストの範囲をフェーズに応じて段階的に拡大する。

#### テスト種別と責務

| テスト種別 | 実行者 | 目的 |
|-----------|--------|------|
| 単体テスト / 結合テスト | AIエージェント | 実装の正しさを検証 |
| セルフレビュー | AIエージェント | コード品質・規約準拠を自己検証 |
| E2Eテスト（自動） | CI / AIエージェント | ユーザーフローの動作保証 |
| E2Eテスト（手動） | 開発者 / QAエンジニア | 自動化困難な操作・UXの確認 |

#### フェーズ別のE2Eテスト範囲

```
Lv.0  スパイク・技術検証
------------------------------------------------------------
  単体テスト    : AIが必要に応じて作成
  E2E（自動）  : なし
  E2E（手動）  : 開発者が動作確認
  * 最低限の動作確認は実施


Lv.1  プロトタイプ・PoC
------------------------------------------------------------
  単体テスト    : AIが実装と同時に作成
  E2E（自動）  : なし（又は主要導線のみ）
  E2E（手動）  : 開発者がPR毎に動作確認
  * 「触って動く」ことを開発者が確認する
  * 自動E2Eの整備コストはかけない


Lv.2  ベータ版・PMF検証
------------------------------------------------------------
  単体テスト    : AIが実装と同時に作成
  E2E（自動）  : 主要ユーザーフローをカバー
  E2E（手動）  : 機能単位でQA実施
  * 主要ユーザーフロー（ログイン、コア機能、決済等）の
    自動E2Eを整備し、CIで毎PR実行
  * 新機能追加時はE2Eテストケースも同時に追加
  * リリース前に機能単位の手動QAを実施


Lv.3  製品版
------------------------------------------------------------
  単体テスト    : AIが実装と同時に作成
  E2E（自動）  : 全ユーザーフローをカバー（回帰テスト含む）
  E2E（手動）  : リリース前にQAエンジニアが網羅的に実施
  * 全ユーザーフローの自動E2Eを整備・維持
  * リグレッションテストスイートをCIで常時実行
  * リリース前にQAエンジニアが探索的テストを含む
    網羅的な手動QAを実施
  * 規制対応がある場合はテストエビデンスを記録
```

#### テストレベル早見表

| | 単体テスト | E2E（自動） | E2E（手動） |
|------|----------|------------|------------|
| Lv.0 | 必要に応じて | なし | 開発者が動作確認 |
| Lv.1 | 必須 | なし〜主要導線 | 開発者が動作確認 |
| Lv.2 | 必須 | 主要ユーザーフロー | 機能単位でQA |
| Lv.3 | 必須 | 全フロー+回帰 | QAエンジニアが網羅的に実施 |

> **原則**: コードレビューで見つけるべきバグは、テストで検出する。設計レビューで方向性を正し、テストで品質を担保する——これがAI駆動開発における品質保証の基本構造である。

### 5.4 クリティカル箇所

以下に該当する場合、**品質保証レベルに関わらず**（Lv.0を除く）設計レビュー・コードレビューの両方を実施する:

- セキュリティに関わる箇所（認証・認可、入力検証）
- 機密データ・個人情報の取り扱い
- 外部システム連携
- データベーススキーマの変更

> **注意**: Lv.0であってもクリティカル箇所を含む変更を本番コードに流用する場合は、適切なレベルで再レビューを行うこと。

## 6. ナレッジ管理

### 6.1 AIエージェントへのコンテキスト共有

AIエージェントが正確に実装するには、十分なコンテキストの提供が不可欠である。以下のドキュメントを整備・維持する:

| ドキュメント | 目的 | 更新頻度 |
|------------|------|---------|
| CLAUDE.md | プロジェクトコンテキスト（規約、構成、コマンド） | 随時 |
| コーディングガイドライン | コード規約の統一 | 随時 |
| アーキテクチャ設計書 | 全体構成の理解 | 機能追加時 |
| API仕様書 | エンドポイント定義 | API変更時 |
| テーブル定義書 | DB構造の理解 | スキーマ変更時 |
| 用語集 | ドメイン知識の共有 | 随時 |

### 6.2 チケットの質がエージェントの質を決める

AIエージェントのアウトプット品質は、**チケットの記述品質**に直結する。

良いチケットの条件:
- 1エージェントが独立して完結できる粒度
- 明確な完了条件（受入基準）
- 参照すべきドキュメント・コードの明示
- 技術的な制約の記載
- 他チケットとの依存関係の明示

詳細は [チケット記述ガイドライン](./workflows/ticket-writing.md) を参照。

## 7. リスクと対策

| リスク | 対策 |
|--------|------|
| AIの出力品質のばらつき | 品質ゲート（自動チェック + 人間レビュー）で担保 |
| AIへの過度な依存 | クリティカル箇所は品質レベルに関わらず人間がレビュー |
| レビュー負荷の集中 | 品質保証レベルでフェーズに応じた負荷コントロール |
| コンテキスト不足による実装ミス | CLAUDE.md・設計書・チケット記述の整備で対処 |
| オーナーシップの属人化 | 他開発者レビュー（Lv.2任意/Lv.3必須）でナレッジ共有 |
| Lv.0の乱用による品質低下 | Lv.0の適用範囲をスパイク・技術検証に限定。本番流用時は再レビュー必須 |
