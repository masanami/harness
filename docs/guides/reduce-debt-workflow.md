# reduce-debt × simplify 統合ワークフロー

`reduce-debt` スキルで技術負債を修正した後、`simplify` スキルを組み合わせることで修正後のコード品質をさらに向上できます。

---

## 全体フロー

```
1. /reduce-debt {親Issue番号}
   ├── スキャン: Agent Teams でプロジェクト全体を並列スキャン
   ├── 結果分類: 今回の実装で導入 / 既存の負債
   └── Issue起票: ユーザー確認後に修正Issueを起票

2. /para-impl {起票されたIssue番号...}
   ├── 技術負債の修正を実装
   ├── テスト・コミット
   └── PR作成

3. /simplify  ← オプション（推奨）
   └── 修正済みコードの品質をさらに向上
       ├── 不要な複雑さの除去
       ├── 重複コードの統合
       └── 可読性・保守性の向上
```

---

## simplify を使うべき場面

### 適用を推奨する場合

| 条件 | 理由 |
|------|------|
| 修正ファイルが5ファイル以上 | 広範な変更ほど一貫性の問題が生じやすい |
| リファクタリング系の修正 | コード構造変更後に冗長なパターンが残ることがある |
| 「コード品質」カテゴリの負債修正 | 重複コード・複雑性の修正と相性が良い |
| 複数人のエージェントが並列修正した場合 | スタイルや命名の不一致が生じやすい |

### 適用を省略できる場合

| 条件 | 理由 |
|------|------|
| セキュリティ・パフォーマンス系のみの修正 | ロジック変更が中心でコードスタイルへの影響が少ない |
| 設定ファイルのみの修正 | simplify の対象外となるケースが多い |
| 修正範囲が1〜2ファイルの軽微な修正 | コストに対して効果が小さい |

## 具体的な実行例

### 例: 親Issue #42 の実装後に技術負債をチェックして修正

```bash
# Step 1: 技術負債スキャン
/reduce-debt 42

# (Agent Teams でスキャン実行、結果レポートを確認)
# → Issue #50: 重複ロジックの統合
# → Issue #51: 過度な複雑性の解消
# → Issue #52: テストカバレッジ不足

# Step 2: 修正の実装
/para-impl 50 51 52

# (修正が実装されPRが作成される)

# Step 3: 修正後の品質向上
/simplify

# → code-simplifier が修正済みコードをレビューし品質を向上
```

---

## 各スキルの役割分担

| スキル | 担当フェーズ | 主な作業 |
|--------|------------|---------|
| `/reduce-debt` | 検出・起票 | 技術負債のスキャン、分類、Issue起票 |
| `/para-impl` | 修正実装 | Issueの実装、テスト、PR作成 |
| `/simplify` | 品質向上 | 修正済みコードの明確さ・一貫性・保守性の向上 |

> **重要**: `simplify` は必ず `/para-impl` による修正実装の後に実行する。スキャン段階や Issue 起票段階では使用しない。
