# ハーネス機能評価フレームワーク

## 1. 概要

本ドキュメントは、harness のスキル・エージェントが「意図通りに動くか」「プロダクトの品質を改善できているか」を検証するためのフレームワークを定義する。

スキルのプロンプト変更時やエージェント構成の変更時に品質劣化を早期検知し、新スキル追加時の評価基準を明確にすることを目的とする。

## 2. 評価の観点・指標

### 2.1 基本評価軸

すべてのスキル・エージェントに共通する評価軸を以下に定義する。

| 評価軸 | 定義 | 測定対象 |
|--------|------|----------|
| **指示への追従率** | スキルの指示書（SKILL.md）に記載された手順を漏れなく実行しているか | 実行ステップの完遂度 |
| **生成物の正確さ** | 出力（コード・ドキュメント・コマンド）が要件を満たしているか | 成果物の品質 |
| **副作用の有無** | スコープ外の変更・不要なファイル追加・意図しない削除が発生していないか | 影響範囲 |
| **エラーハンドリング** | 異常系・境界条件で適切に振る舞うか | 堅牢性 |
| **ユーザーへの確認タイミング** | 確認すべき場面で確認し、不要な場面では自律的に進むか | インタラクション品質 |

### 2.2 スキル固有の評価指標

スキルの特性に応じて、以下の追加指標を設定する。

#### 実装系スキル（para-impl, compare-impl, auto-develop）

| 指標 | 定義 |
|------|------|
| TDD遵守 | テストを先に書いてから実装しているか |
| コミット粒度 | Conventional Commits形式で適切な粒度でコミットしているか |
| PR品質 | PR本文に `Closes #番号` が含まれ、実装内容が明確に記載されているか |
| ブランチ命名 | `feature/issue-{番号}-{説明}` の命名規則に従っているか |

#### レビュー系スキル（self-review, code-reviewer, design-reviewer）

| 指標 | 定義 |
|------|------|
| 観点網羅性 | SKILL.mdに定義されたすべての観点でレビューしているか |
| 誤検知率 | 問題のないコードを誤って指摘していないか |
| 見逃し率 | 明らかな問題を見逃していないか |
| 報告形式 | 問題点・改善提案・良い点の形式で出力しているか |

#### 分析系スキル（reduce-debt, evaluate-candidates）

| 指標 | 定義 |
|------|------|
| 検出精度 | 存在する問題を適切に検出しているか |
| 優先度の妥当性 | 高/中/低の優先度付けが基準に照らして妥当か |
| Agent Teams提案の適切さ | 適切な粒度でteammateを分割しているか |

### 2.3 合格基準の目安

| 評価軸 | 目標値 |
|--------|--------|
| 指示への追従率 | 主要ステップを100%完遂、オプション手順は90%以上 |
| 生成物の正確さ | 要件を満たす成果物を生成 |
| 副作用の有無 | スコープ外変更ゼロ |
| ユーザーへの確認タイミング | 確認必要場面での確認漏れゼロ、不要な確認は最小化 |

## 3. 評価シナリオの設計方法

### 3.1 シナリオの構成要素

各評価シナリオは以下の要素で構成する。

```markdown
## テストケース
### TC-{番号}: {ケース名}
- **前提**: {セットアップ条件（リポジトリ状態、Issue内容等）}
- **入力**: {スキルへの入力（コマンド・引数）}
- **期待する動作**: {期待する振る舞いを具体的に記述}
- **合格基準**: {具体的・客観的な合格条件（YESかNOで判定可能なもの）}
- **チェック方法**: {合否の確認手順（コマンド・目視確認等）}
```

### 3.2 ケース分類

各スキルのシナリオは以下の分類でカバーする。

| 分類 | 内容 |
|------|------|
| **正常系** | 典型的なユースケース。スキルが意図通りに動くことを確認 |
| **境界条件** | 入力が空・最小・最大の場合 |
| **異常系** | 不正な入力・存在しないIssue番号等 |
| **副作用確認** | スコープ外変更が発生していないことを確認 |

### 3.3 シナリオ設計の原則

- **再現可能**: 誰が実行しても同じ手順で評価できること
- **客観的**: 合否が「YESかNO」で判定できること（主観に頼らない）
- **独立**: 他のテストケースの結果に依存しないこと
- **最小**: 1ケースで1つの観点を評価すること

## 4. 評価実行方法

### 4.1 実行方式

本フレームワークは以下の3方式で評価を実行する。

#### 方式A: 手動評価（基本）

評価者がシナリオ手順に従って手動でスキルを実行し、合否を判定する。

- **対象**: すべてのシナリオ
- **タイミング**: スキルのプロンプト変更時・エージェント構成変更時
- **実施者**: 開発者・レビュアー

```bash
# 評価実行の基本フロー
# 1. 評価用リポジトリのセットアップ
# 2. シナリオの前提条件を整備
# 3. スキルを実行
# 4. 合格基準に照らして合否を判定
# 5. 結果をeval/results/{日付}/{スキル名}.mdに記録
```

#### 方式B: 半自動評価

シナリオの中で自動検証可能な部分（コマンド実行・ファイル存在確認等）をスクリプト化する。

- **対象**: チェック方法にコマンドが明記されているケース
- **タイミング**: 方式Aと同様
- **実施者**: 開発者（スクリプト実行）

評価スクリプトの例:
```bash
# eval/scripts/check-branch-name.sh
# ブランチ命名規則チェック
BRANCH=$(git branch --show-current)
if echo "$BRANCH" | grep -qE '^(feature|fix)/issue-[0-9]+-'; then
  echo "PASS: ブランチ命名規則OK ($BRANCH)"
else
  echo "FAIL: ブランチ命名規則違反 ($BRANCH)"
fi
```

#### 方式C: CI連携（将来対応）

GitHub Actionsを用いてPR時に自動評価を実行する。

- **対象**: スキルのプロンプトファイル変更を含むPR
- **タイミング**: 自動（PR作成時・更新時）
- **実施者**: CI

```yaml
# .github/workflows/eval.yml（将来実装予定）
on:
  pull_request:
    paths:
      - 'skills/**'
      - 'agents/**'
```

### 4.2 評価記録

評価結果は `eval/results/{YYYY-MM-DD}/{スキル名}.md` に記録する。

```markdown
# {スキル名} 評価結果 {YYYY-MM-DD}

## 評価環境
- 実施日: {日付}
- 評価者: {氏名}
- 対象バージョン: {コミットハッシュ}

## 結果サマリー

| ケース | 合否 | 備考 |
|--------|------|------|
| TC-01 | PASS | - |
| TC-02 | FAIL | {失敗理由} |

## 詳細

### TC-01: {ケース名}
- 合否: PASS
- 実行ログ: {ログ or スクリーンショットへのリンク}
- 所見: {気づき・改善提案}
```

### 4.3 評価のトリガー

| トリガー | 対象スキル | 実施方式 |
|----------|-----------|----------|
| スキルのプロンプト変更（SKILL.md更新） | 変更対象スキル | 方式A |
| エージェント定義変更（agents/*.md更新） | 関連スキル全体 | 方式A |
| 新スキル追加 | 追加スキル | 方式A（全ケース） |
| ハーネス全体のメジャーアップデート | 全スキル | 方式A |

## 5. 新スキル追加時の評価基準

新しいスキルを追加する際は、以下の評価基準を満たすシナリオを整備してからリリースする。

### 5.1 必須評価ケース

| # | 分類 | 内容 |
|---|------|------|
| 1 | 正常系 | 典型的なユースケースで意図通りに動作すること |
| 2 | 副作用確認 | スコープ外の変更が発生しないこと |
| 3 | 確認タイミング | SKILL.mdに「ユーザーへの確認タイミング」が定義されている場合、その場面で確認すること |

### 5.2 推奨評価ケース

| # | 分類 | 内容 |
|---|------|------|
| 4 | 異常系 | 不正入力・存在しないリソースへの対応 |
| 5 | 境界条件 | 最小・最大入力での動作 |
| 6 | 連携確認 | 他スキル・エージェントとの連携が正しく機能すること |

### 5.3 シナリオファイルの配置

新スキルのシナリオは `eval/scenarios/{スキル名}.md` に配置する。フォーマットは既存シナリオ（[self-review](../eval/scenarios/self-review.md) 等）に準拠する。

## 6. フレームワークの改善方針

### 6.1 シナリオのメンテナンス

- スキルのプロンプト変更時に、対応するシナリオも同時に更新する
- 評価実施時に発見した新たな観点は、シナリオに追加する
- 半年に1回、シナリオ全体を見直してフレームワークの有効性を評価する

### 6.2 指標の精緻化

評価の実施を重ねる中で、以下を継続的に改善する:

- 合格基準の閾値の妥当性を検証し、必要に応じて調整する
- 誤検知・見逃しが多い評価観点は、定義を見直す
- 自動化可能な観点は方式Bまたは方式Cに移行する
